# 배포 전 QA 체크리스트

> 이걸 끝까지 통과해야 배포. 각 시나리오마다 ✅ 체크박스 + 발견된 버그는 하단 "이슈 로그"에 기록.

**테스트 환경 권장:**
- 데스크톱: Chrome (필수), Safari (Mac 있으면), Firefox
- 모바일: iPhone Safari + Android Chrome (실기기 권장)
- 계정 최소 3개:
  - **A**: 호스트 (이벤트 만드는 사람)
  - **B**: 신청자 (RSVP 보내는 사람)
  - **C**: 비로그인 게스트

---

## 1. 인증 / 온보딩

### 1.1 첫 로그인 (신규 유저)
- [ ] `/login`에서 Google 로그인 클릭 → Google 인증 → `/onboarding`으로 자동 이동
- [ ] 온보딩 폼에 Google 이름이 미리 채워져 있음
- [ ] 프로필 사진 — Google 사진이 기본 표시
- [ ] "Reset to Google photo" 버튼 작동
- [ ] 사진 직접 업로드 후 "Reset" 누르면 다시 Google 사진으로 돌아감
- [ ] 이름, 국적, 신분 필수 입력 시 비우면 제출 안 됨
- [ ] 소셜 링크 **최소 1개 이상 필수**
- [ ] 소셜 링크 잘못된 URL (예: "abc")이면 에러 메시지
- [ ] 제출 성공 시 `/meetups`로 이동

### 1.2 기존 유저 재로그인
- [ ] 로그아웃 → 다시 로그인 → 온보딩 거치지 않고 바로 `/meetups`
- [ ] 프로필 사진/이름이 DB에 저장된 값으로 표시 (Google 사진이 아님)

### 1.3 온보딩 미완료 유저
- [ ] DB에서 강제로 `onboarding_complete=false`로 만들고 `/profile` 접근 → `/onboarding`으로 리다이렉트
- [ ] `/meetups`는 접근 가능 (비로그인 + 미완료 둘 다)

### 1.4 비로그인 (게스트)
- [ ] `/`, `/meetups`, `/feed` 접근 가능
- [ ] 미팅 상세 페이지 접근 가능, "Who's going" 자리에 "Sign in to see who's going" 표시
- [ ] RSVP 버튼 클릭 → `/login?from=...`로 이동
- [ ] 로그인 후 원래 페이지로 복귀

  - [ ] 클릭하면 파일 선택 다이얼로그

## 6. RSVP (참가 신청)

### 6.1 신청
- [ ] 비로그인 → 로그인 페이지로
- [ ] 로그인 + 미신청 → "Request to Join" 버튼
- [ ] 클릭 → 메시지 입력 다이얼로그
- [ ] 메시지 비워도 신청 가능
- [ ] 신청 후 버튼이 "Pending" 등 상태 표시로 변경
- [ ] DB에 RSVP 정상 저장 (`status='pending'`)

### 6.2 호스트 - Pending 처리
- [ ] 호스트가 자기 이벤트 상세 페이지 → "Pending Requests" 섹션에 신청자 표시
- [ ] 신청자 프로필 사진, 이름, 국적, bio 정상 표시
- [ ] 신청 메시지 인용구 스타일로 표시
- [ ] **Approve** 클릭 → 즉시 처리, 페이지 새로고침되며 "Done" 또는 사라짐
- [ ] **Decline** 클릭 → 즉시 처리
- [ ] 다 처리하면 "Pending Requests" 섹션 자체가 사라짐
- [ ] 신청자 입장에서 페이지 새로고침 → 버튼이 "Approved" / "Rejected"로 바뀜

### 6.3 만석 처리
- [ ] 정원이 다 차면 RSVP 버튼이 "Full" 비활성 상태로
- [ ] 호스트가 누군가를 거절하면 다시 활성화
- [ ] 호스트는 만석이어도 본인이 참석자 아니므로 RSVP 버튼 안 보임 (자기 이벤트라서)

### 6.4 비정상 케이스
- [ ] 같은 사람이 두 번 신청 시도 → 거부 또는 idempotent
- [ ] 본인이 자기 이벤트에 RSVP → UI에서 막혀있어야 함
- [ ] 지난 이벤트에 RSVP 시도 → "This meetup has ended" 표시, RSVP 버튼 자체 없음

## 8. Memories (이벤트 후기)

### 8.1 노출 조건
- [ ] **이벤트 날짜 지난 후에만** Memories 섹션 노출
- [ ] 미래 이벤트 페이지엔 Memories 섹션 없음
- [ ] 호스트 또는 approved 참석자만 입력창 보임
- [ ] 기타 사용자 / 비로그인은 읽기만 가능

### 8.2 작성
- [ ] 글만 — 사진 없이 제출 가능
- [ ] 사진만 — 글 없이 제출 가능
- [ ] 둘 다 비우면 제출 불가
- [ ] 사진 1장, 2장, 3장+ 각각의 그리드 레이아웃 정상
- [ ] 사진 최대 10장 제한 (11장째 업로드 막힘)
- [ ] 사진 미리보기 ✕로 개별 제거
- [ ] 1000자 초과 시 슬라이스되거나 막힘
- [ ] 제출 성공 → 즉시 리스트 상단에 추가

### 8.3 조회
- [ ] 작성 시간 ("just now", "5m ago", "Apr 12" 등) 정상
- [ ] 사진 클릭 → 라이트박스 열림
- [ ] 라이트박스에서 ← / → 키로 사진 전환
- [ ] Esc 키로 닫힘
- [ ] 배경 클릭으로 닫힘

### 8.4 삭제
- [ ] 본인 메모리에만 ✕ 버튼 보임
- [ ] 삭제 확인 다이얼로그 → 확인 시 즉시 사라짐

## 9. Feed (자유 게시판)

### 9.1 게시글
- [ ] `/feed` 진입 → 최신순 정렬
- [ ] 비로그인이면 "Share something..." 입력창 안 보임
- [ ] 로그인 후 입력창 포커스 시 행 늘어남, Cancel/Post 버튼 노출
- [ ] 빈 글 제출 불가
- [ ] 1000자 초과 슬라이스
- [ ] 제출 → 리스트 최상단에 추가

### 9.2 좋아요
- [ ] 빈 하트 클릭 → 즉시 채워지고 카운트 +1 (optimistic)
- [ ] 다시 클릭 → 빈 상태로, 카운트 -1
- [ ] 페이지 새로고침 후 상태 유지
- [ ] 비로그인은 좋아요 비활성

### 9.3 댓글
- [ ] 댓글 아이콘 클릭 → 토글 열림
- [ ] 비로그인엔 입력창 안 보임
- [ ] 댓글 작성 → 즉시 추가, 카운트 +1
- [ ] 비어있을 땐 빈 상태 ("Be the first to comment" 등)
- [ ] 본인 댓글에만 ✕ 버튼

### 9.4 대댓글
- [ ] 댓글에 "Reply" 버튼 → 인라인 입력창
- [ ] 대댓글 작성 → 부모 댓글 아래에 들여쓰기로 표시
- [ ] **대댓글에 또 Reply** → 같은 부모 아래에 평탄화 (트리 깊이 1단계 유지)
- [ ] 대댓글 삭제 → 부모 댓글은 그대로 유지

### 9.5 게시글 삭제
- [ ] 본인 글에만 ✕ 버튼
- [ ] 삭제 확인 → 즉시 리스트에서 사라짐, 댓글/좋아요도 cascade로 사라짐

## 13. 에러 핸들링

### 13.1 404
- [ ] 존재하지 않는 미팅 ID (`/meetups/abc123`) → notFound() 트리거 → Next 기본 404 페이지

### 13.2 백엔드 다운
- [ ] 백엔드 stop → 페이지 로드 시 어떻게 되는지 (현재는 try/catch로 빈 배열 처리)
- [ ] 백엔드 다시 띄우면 자동 복구

### 13.3 권한 에러
- [ ] 비호스트가 `/host/events/[id]/edit` → 리다이렉트
- [ ] 비호스트가 `PATCH /api/host/events/[id]` 직접 호출 → 403
- [ ] 비참석자가 `POST /api/events/[id]/memories` → 403

- [ ] 텍스트 잘림 없음
- [ ] 이미지 깨짐 없음
- [ ] 터치 타겟 충분히 큼 (최소 44x44px)

---

## 15. 성능

- [ ] 첫 페이지 로드 (캐시 없음, 4G 시뮬레이션) — 3초 이내
- [ ] 이미지 많은 페이지 (Memories) 부드럽게 스크롤
- [ ] 라이트박스 열기/닫기 부드러움
- [ ] 좋아요 클릭 즉시 반응 (optimistic update)
- [ ] DevTools Network에서 API 호출 한 번에 끝나는지 (waterfall 비효율 X)

---

## 16. 접근성 (Basic)

- [ ] 모든 인터랙티브 요소가 키보드로 접근 가능 (Tab으로 순회)
- [ ] 폼 input에 label 연결 (스크린리더가 읽음)
- [ ] 이미지에 alt 속성 (장식적인 건 빈 alt OK)
- [ ] 색만으로 정보 전달 안 함 (예: 상태 배지에 색 + 텍스트 모두)

---

## 17. 보안 (QA에서 빠르게 점검 가능한 것)

- [ ] DevTools Console에 시크릿/토큰 출력 X
- [ ] DevTools Network에서 다른 사용자 ID로 mutation 시도 → 403
- [ ] HTML 소스에 `NEXTAUTH_SECRET`, `GOOGLE_CLIENT_SECRET` 등 노출 X
- [ ] XSS 시도 (게시글에 `<script>alert(1)</script>` 입력) → 텍스트로만 표시, 실행 X
- [ ] HTML/Markdown 시도 (`**bold**`, `<b>x</b>`) → 그대로 텍스트 표시 (의도 — escape 됨)

---

## 18. 데이터 정합성

- [ ] 사용자 삭제 시 그 사람의 RSVP/post/comment/memory가 cascade로 정리되는지 (DB FK 확인)
- [ ] 이벤트 삭제 (수동 SQL) 시 그 이벤트의 RSVP/memory/comment가 cascade로 정리되는지
- [ ] 게시글 삭제 시 댓글/좋아요 cascade
- [ ] 댓글 삭제 시 대댓글 cascade

---

## 19. 시드 / 데모 데이터

배포 후 첫 사용자가 와서 빈 화면 보지 않게:
- [ ] 데모 미팅 1~2개 미리 만들어둘지 결정
- [ ] 또는 빈 상태 그래픽이 충분히 친근한지 (사용자 모집까진 견딜만 한지)
